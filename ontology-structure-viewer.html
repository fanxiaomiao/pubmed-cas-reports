本体结构说明

本页面呈现微生物组文献知识库的本体（ontology）结构，用于为文献与专利中抽取的实体——菌株、代谢物、宿主生物过程、健康表型等——及其相互关系提供统一的语义定义，作为下游知识图谱的 schema 层。Schema 确定后，无论是人工标注还是 LLM 自动抽取，产出数据均遵循同一套受控词表与关系约束，具备跨文献聚合、路径查询及部分自动推理能力。

建议浏览顺序：网络图展示证据来源（文献/专利）与各语义分支的连接关系；类层级可查阅具体类目的定义与说明；属性一览列出全部对象/数据属性及其逻辑特性（FunctionalProperty、TransitiveProperty、SymmetricProperty 等），悬停徽标查看简要说明，不要求预先具备描述逻辑（Description Logic）背景。

发现建模问题或有修改意见，可记录在右下角备注栏，支持一键复制导出；内容保存在本机浏览器 localStorage，不做跨设备同步。

建模范围与已知局限

以下几处是当前设计中存在多种合理实现方式、尚未定论的建模选择，其余细节性设计不在此列出。

①
综述类文献是否应约束为不含实验数据
综述（Review）类文献在语义上通常不涉及原创实验，但当前 schema 未对此施加硬性约束（即未声明 ReviewArticle 与 hasExperiment/hasBioinformaticsAnalysis 的不相容公理）。加约束的收益是可自动拦截"综述被误标注实验数据"的录入错误；代价是若某综述确实包含作者补充的原创再分析，会被误判为不一致。
②
体内实验动物物种与宿主类型（HostType）词表尚未统一
"体内实验所用动物物种"与"宿主类型"（人/小鼠/果蝇……）目前是两套独立的自由文本字段，未做关联。统一后可支持"检索所有以小鼠为模型的体内实验"这类跨字段查询；代价是需要额外的建模工作，且历史数据若两侧填写口径不一致，统一后可能暴露既有数据的规范性问题。
③
ResearchArticle 与 ClinicalTrialArticle 是否合并
当前两者作为独立的互斥子类分别管理，粒度更细；若业务查询只关心"研究型 / 综述 / 其他"的粗粒度划分，合并为单一类别会更简洁，但会损失临床试验这一细分标签。

文献 / 专利证据网络图

以 EvidenceSource（证据来源，Paper 与 Patent 的共同上位类）为中心，展开其一跳可达的全部 32 个类节点（含 Author 作者实体、HealthCondition 细分表型）。灰色实线为正向抽取关系；蓝色实线为声明了 owl:inverseOf 的双向可查关系（如 hasPhenotype ↔ isPhenotypeOf）；虚线为 rdfs:subClassOf；黄色虚线为 involvesStrain 属性链（owl:propertyChainAxiom）——由功能方向经证据来源推理直达关联菌株，属推理产生的隐含关系，非人工断言。节点支持拖拽重新布局与滚轮缩放，悬停节点高亮其关联边。

推理定义类 Reasoner-inferred / constrained classes

此处类别不依赖人工标注，而是由 DL 推理器根据逻辑定义自动判定成员资格，对应两类公理：充分必要条件（owl:equivalentClass，满足条件即被自动分类）与必要条件（rdfs:subClassOf + owl:Restriction，隶属该类必须满足条件，但满足条件不蕴含隶属关系，主要用于数据一致性校验）。两类公理均已通过 HermiT 推理器实测验证（含正向分类与反向不一致检测）。

类层级

按一级语义分支分组，点击展开子类；点击类名查看定义说明及相关属性。斜体条目为该类下的具体个体（instance），种/株级个体按学名命名规范排版；橙色「示例」标记为验证推理用的演示数据，非正式语料。

    属性一览

    domain → range 表示该关系的定义域与值域。三类颜色徽标对应属性的逻辑特性：蓝色「F」= FunctionalProperty，同一主语至多一个取值，用于数据一致性校验（如一篇文献不应有两个不同的 DOI）；绿色「T」= TransitiveProperty，关系沿链条自动传递（如分类学阶元"界→门→纲"逐级断言后，可直接推出"界"与某具体菌株的隶属关系，无需逐级连接）；紫色「S」= SymmetricProperty，关系双向对称，断言一次即隐含反向关系成立（如作者合作网络中 A collaboratesWith B 可推出 B collaboratesWith A）。
